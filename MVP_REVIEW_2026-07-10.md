# OneDesk MVP 复盘与 v0.2 改进建议

日期：2026-07-10  
范围：`PRD.md`、`DEV_PLAN.md`、两份原始小红书岗位 PRD、当前 `app/` 可运行 MVP、原始界面规划图，以及 Claw-Empire / My Virtual Office 的产品表达方式。

## 1. 结论先行

OneDesk 当前 MVP 已经具备一条完整、克制且可信的业务骨架：公司知识 → 内容草稿 → 人工审批与发布 → 评论导入 → 线索判断 → 人工跟进 → 数据复盘。它不是一个“演示用聊天机器人”，而是一个可以承载真实业务数据的本地工作台。

但现在产品价值仍被三个缺口限制：

1. **线索来源仍然依赖用户手动搬运。** 系统会判断，但还没有把“每天刷小红书、复制评论”这段高成本工作显著缩短。
2. **Agent 的工作过程不够可见。** 当前有结果、有理由，但缺少运行计划、使用了哪些资料、遇到什么阻塞、下一步等待谁的更清晰表达。
3. **首页像后台，不像一家公司。** 原有办公室页信息正确，但员工、目标、审批、流程彼此分散，用户看不到“两位员工正在一起完成一件事”的现场感。

v0.2 不应扩展到更多行业、更多员工或员工商店。应集中把“财税公司老板每天少刷 1 小时、得到 10 条可判断线索、20 分钟内完成确认”做实。

## 2. 值得保留的产品判断

- 单用户、本地优先、SQLite：适合早期真实试用，也降低客户资料与平台登录信息的顾虑。
- 内容与线索双员工，而不是通用 Agent：用户很容易理解职责、产出和边界。
- 人工发布、人工回复：这是风控能力，不是自动化不足。
- 每条线索保留来源、原文、评分和理由：这是长期建立信任、评估模型质量的基础。
- 知识完整度门槛：能阻止模型在公司事实不足时编造业务口径。
- 演示模式与真实 LLM 双通道：便于销售演示和无 Key 首次体验。

## 3. 当前流程的关键改进

### 3.1 首次使用：从“填资料”改为“20 分钟开工诊断”

当前知识库逻辑正确，但填写体验容易让老板觉得是在替系统做实施工作。建议改成四步引导：

1. 公司与城市：公司名、城市、服务范围。
2. 三个主推服务：周期、价格区间、所需材料、不能承诺的事项。
3. 真实销售口径：导入已有介绍文档，或粘贴 5 组常见问答。
4. 开工体检：系统列出“可以说 / 暂时不能说 / 仍缺资料”，用户确认后解锁员工。

验收指标：首次注册到生成第一批可审内容，目标小于 20 分钟；必填字段放弃率小于 30%。

### 3.2 内容：从“每天生成 3 篇”改为“每天做 1 个可验证实验”

三篇内容不应该只是三个独立文本。每批内容应绑定同一个增长假设：

- 目标人群：电商个体户 / 初创公司 / 自由职业者；
- 需求场景：注册、记账、注销、异常解除；
- 搜索意图：怎么办、多少钱、避坑、材料清单；
- 预期动作：收藏、评论咨询、进入主页；
- 本次只变化的变量：标题角度、封面表达或内容结构。

发布后，用户只需回填链接和基础数据。周报按“假设是否成立”复盘，而不是只罗列点赞数。

### 3.3 线索：优先解决来源摩擦，而不是先提高模型复杂度

建议将线索入口按优先级分为：

1. **自家笔记评论**：最高优先级、意图最强、风险最低；发布后登记链接，首页持续提醒导入评论。
2. **用户主动打开的搜索结果/帖子**：提供“采集当前页面”按钮，由本地执行器读取当前可见公开内容。
3. **用户选择的关键词观察清单**：本地执行器在用户已登录且页面正常时做低频、只读、可中断的巡检。

每次采集必须记录：触发人、关键词、页面 URL、采集时间、可见原文、失败原因。验证码、登录失效或风控提示出现时立即暂停，并把界面交还给用户。

### 3.4 判断与跟进：把评分变成可校准的销售助手

- 评分拆成 4 个可解释维度：需求明确度、时间紧迫度、地域/服务匹配、行动信号。
- reason 必须逐项引用评论原文，不能只给总结性判断。
- 用户“忽略 / 通过 / 已回复 / 已转化”的结果反哺评分规则；先做可见的规则权重，不急于自动训练模型。
- 增加“为什么推荐这句话”与命中的知识条目，避免回复看似自然但违背公司口径。
- 记录“联系上 / 有需求但不匹配 / 暂无预算 / 已成交 / 错误线索”，让周报能够计算真正的线索质量。

### 3.5 每日首页：只回答三个问题

1. 两位员工现在在做什么、为什么停下；
2. 今天最需要老板做的一个动作是什么；
3. 内容到线索的业务链路走到了哪一步。

本次界面改造已经按这个原则重组办公室页：员工进入可视办公室场景，今日目标、员工状态、待确认、运行记录和获客链路被放到同一个叙事中。保留纸感、墨色、朱砂和宋体标题，没有引入积分、货币、等级或宠物。

## 4. Agent 能力的正确边界

OneDesk 需要的是**有证据、可暂停、受监督的业务 Agent**，不是在小红书上模拟真人的全自动机器人。

```text
老板设定目标与边界
        ↓
编排层：拆任务、选择知识、生成计划、记录状态
        ↓
本地执行层：读取本地文件 / 当前浏览器可见页面 / 生成文本
        ↓
证据层：保存来源、原文、模型理由、合规扫描、运行日志
        ↓
审批层：老板确认发布、回复、私信等高风险动作
        ↓
结果层：回填真实发布与跟进结果，驱动下一轮内容和评分
```

### 可以逐步自动化

- 整理公司资料、检测缺失与冲突；
- 生成内容计划、草稿、发布清单和封面文字；
- 读取用户主动打开页面中的公开帖子/评论；
- 去重、分类、评分、生成判断理由与拟回复；
- 生成每日简报和周复盘；
- 监控任务失败、登录过期、验证码和数据异常并提醒用户。

### 始终需要人工确认

- 发布笔记；
- 发送评论或私信；
- 关注、点赞或其他会改变平台状态的动作；
- 任何报价承诺、税务结论、案例或客户隐私相关内容；
- 验证码、账号登录与风控处理。

## 5. 最合适的产品形态

结论：**近期是“本地 Web 工作台”，下一步是“Web 工作台 + 轻量本地执行器”的混合形态；不建议现在重写成纯桌面应用，也不建议直接做纯云 SaaS。**

### 为什么不是纯云 SaaS

- 小红书登录、验证码和浏览器状态天然在用户设备上；
- 财税公司资料、销售话术、客户评论具有隐私敏感性；
- 云端主动采集更容易形成账号、IP 和平台规则风险。

### 为什么暂时不做纯桌面应用

- 当前主要交互是表格、审批、编辑与数据复盘，Web UI 的迭代与测试成本更低；
- MVP 还在验证“每天是否愿意用”，此时投入系统级桌面包装收益有限；
- 未来团队协作、远程查看和更新仍需要 Web 控制面。

### 推荐演进

- **v0.1（当前）**：本机启动的 Web 工作台 + SQLite + 手动导入。
- **v0.2**：增加浏览器扩展或本地伴侣进程，实现用户触发的“采集当前页”、任务托盘、登录/验证码暂停提示。
- **v0.3**：将本地 Web 工作台包装为 Tauri/Electron 桌面壳，提供一键安装、自动更新、托盘运行与本地文件权限；业务 UI 仍复用 Web 前端。
- **试点验证后**：云端只负责账号、配置同步、模型网关和远程查看；平台登录态、采集与敏感数据继续留在本地执行面。

## 6. 从参考项目吸收什么、不吸收什么

### Claw-Empire

吸收：local-first、员工状态可见、任务生命周期、运行日志、统一控制面。  
不吸收：多部门、多模型编排、XP 排名、技能商店、通用自治公司。它们会让财税老板看不懂“今天能得到什么”。

### My Virtual Office

吸收：员工在工作/等待/阻塞时有不同可见状态；点击员工可看任务和工具记录；本地执行面需要显式安全边界。  
不吸收：办公室编辑器、家具布置、宠物、天气和自由漫游。OneDesk 的办公室是业务状态的空间化表达，不是一个独立模拟游戏。

两个参考项目都不应直接复制代码或资产。尤其 My Virtual Office 使用 AGPL-3.0，若修改并通过网络向用户提供服务会带来对应开源义务；OneDesk 只借鉴交互原则和产品表达。

## 7. v0.2 优先级

### P0：两周内完成

- 20 分钟首次开工引导和资料体检；
- 每条内容/线索的“使用了哪些知识、为什么这样判断”抽屉；
- 发布链接登记后自动标记 `own_post`，自家评论优先处理；
- 跟进结果细分与可纠错评分；
- 首页下一步建议、员工阻塞原因、运行记录；
- 事件埋点：首次价值时间、生成→通过率、导入→有效线索率、回复→转化率。

### P1：随后两周

- 本地“采集当前页面”原型，只读、用户触发、有限字段；
- 登录失效、验证码和风控暂停机制；
- 任务运行详情：计划、输入、证据、耗时、失败与重试；
- 每日简报和周复盘直接给出下周继续写/停止写清单。

### P2：完成 3–5 家真实客户试点后再决定

- 桌面壳、托盘运行、自动更新；
- 私有云同步与团队查看；
- 低频关键词巡检；
- 企业微信提醒。

## 8. MVP 关键验收指标

- 首次价值时间：从启动到第一批可审内容 < 20 分钟；
- 每日操作时长：内容确认 + 线索确认合计 < 20 分钟；
- 内容可用率：用户轻改或直接通过 ≥ 70%；
- 线索有效率：用户确认的中/强意向占入库线索 ≥ 35%；
- 理由可信率：抽检中 reason 能引用有效原文 ≥ 95%；
- 跟进率：已确认强意向线索 24 小时内完成一次人工跟进 ≥ 70%；
- 闭环使用：连续两周完成“发布—评论—线索—复盘”的试点客户 ≥ 3 家；
- 安全红线：自动发布、自动回复、自动私信、验证码绕过事件 = 0。

## 9. 本次实际改造

- 办公室首页改为“数字员工真实工作现场”：新增专用办公室插画、今日目标、双员工实时状态卡、下一步建议、待确认、运行记录与完整获客链路；
- 左侧导航由文本符号统一替换为 Lucide 图标，并增加明确的本地工作区与数据边界提示；
- 保留原有纸感、墨色、朱砂、苔绿和衬线标题体系；
- 没有引入积分、排名、虚拟货币、宠物、家具编辑器或不必要的通用 Agent 能力；
- 办公室内所有状态和主要入口仍连接真实 API 与现有页面，不是静态展示稿。

## 10. 动态办公室增量设计

办公室不应持续播放“忙碌动画”。持续循环会让用户无法分辨 Agent 是否真的有任务，也会削弱产品信任。动态表达应与任务状态绑定：

- `idle`：人物保持安静，员工卡显示待命；
- `working`：出现输入节奏、当前任务说明和员工卡高亮；
- `waiting_approval`：停止工作动效，朱砂色提示用户确认；
- `done`：显示今日完成和真实产出数量；
- `blocked`：不播放工作动效，说明缺少的公司资料；
- `failed`：任务页保留失败原因和重试入口。

本次已实现状态驱动的轻动效、3 秒状态同步、任务完成后的自动状态切换，以及“内容策划 → 内容确认 → 人工发布 → 评论导入 → 线索判断 → 人工跟进 → 周度复盘”的真实步骤判断。

现有办公室人物仍属于一张完整背景图，因此不能只移动手臂而保持背景完全稳定。真正的逐帧打字需要新增独立人物动画素材层；应在 3–5 家试点客户验证动态状态确实提升理解和使用频率后，再投入 2D sprite、Lottie 或短循环 WebM 素材制作。
