# OneDesk MVP PRD — 小红书财税获客双员工工作台

版本：v0.1（MVP）
日期：2026-07-09
关联文档：`VISION.md`、`小红书内容运营专员 MVP PRD.md`、`小红书获客专员MVP PRD.md`

---

## 1. 产品定位

**一句话**：OneDesk 第一版是一张给财税服务商老板的"数字办公桌"——桌上坐着两位数字员工，一位每天写好 3 篇可发布的小红书笔记，一位每天整理好 10-20 条带判断理由的潜在客户线索，老板只做确认和跟进。

**目标用户**（第一版只服务这一类人）：
- 财税/代账/公司注册服务商的老板或销售，1-10 人团队；
- 有小红书账号但不会运营，或每天人工刷小红书找客户；
- 城市级本地生意（如"东莞注册公司"），关键词高度可枚举。

**核心价值**：
1. 把"今天发什么"变成"今天确认哪 3 篇"——消灭内容焦虑；
2. 把"刷 2 小时小红书找客户"变成"看 10 分钟带评分和拟回复的线索表"；
3. 内容 → 评论 → 线索 → 反馈 → 下一批选题，形成第一个可复盘的获客闭环；
4. 所有判断可追溯：每条线索有来源链接和判断理由，每篇内容有参考爆款。

**非目标（第一版明确不做）**：
- 不做自动私信、批量关注/评论/点赞，不绕过小红书风控，不做账号池；
- 不做多平台（抖音、视频号等）、不做多行业模板、不做数字员工商店；
- 不做复杂 CRM（线索只有状态流转，没有商机金额、合同、回款）；
- 不做多用户/团队权限，第一版单用户本地运行；
- 不做全自动发布，发布永远由人完成；
- 不做大规模爬虫。第一阶段线索和爆款以"手动/半自动导入"为主，轻量采集放第二阶段。

## 2. 第一版信息架构

导航共 8 个区域，左侧固定侧栏，克制、不游戏化：

```
OneDesk
├── 办公室（首页）    今日工作台：两位员工状态、待确认事项、今日产出摘要
├── 员工              两位数字员工的档案页：职责、规则、今日任务、历史产出
├── 任务              任务列表：内容任务 / 线索任务，状态流转，手动触发"开工"
├── 线索              线索池：列表 + 详情，评分筛选，拟回复，跟进状态
├── 内容              内容库:选题库、草稿、待发布、已发布、发布清单导出
├── 数据              复盘看板：发布数据录入、内容-线索归因、周报
├── 知识库            公司知识 / 服务知识 / 销售知识 / 平台知识 / 合规禁区
└── 设置              公司资料、LLM 配置、工作区目录、推送配置（预留）
```

原则：
- "办公室"是唯一入口页，回答"今天我要做什么"，用户每天只需在这一页 + 审批动作里工作；
- "审批中心"不做独立导航，收敛为办公室首页的"待你确认"队列（见第 7 节），减少页面跳转；
- 所有列表页都能看到"来源"和"理由"，这是产品的信任底座。

## 3. 两位数字员工

### 3.1 小红书内容运营专员（代号：阿禾）

| 项 | 内容 |
|---|---|
| 职责 | 研究爆款 → 生成选题 → 生成笔记（标题/封面文字/正文/标签/评论引导）→ 输出发布清单 → 每周复盘 |
| 输入 | 知识库（公司/服务/销售/平台/合规）、关键词任务配置、参考爆款库（ReferencePost）、上周复盘结论、线索反馈（哪些内容带来咨询） |
| 输出 | 每日 10-20 个选题（入选题库）、每日 3 篇 ContentDraft（每篇含 3 版标题以上）、发布清单（可复制/导出）、每周 WeeklyReview |
| 状态 | idle（待命）→ working（生成中）→ waiting_approval（等确认）→ done（今日完成）→ blocked（缺资料/缺 Key） |
| 禁止行为 | 不编造客户案例和数据；不承诺退税/免税/保过等结果；不搬运原文洗稿；不写"最便宜/绝对/100%"类违规词；不生成诱导站外私信的话术；正文不超过 800 字 |
| 质量规则 | 每篇必含：1 个明确痛点、1 个行动引导、3-6 个小节、每段 ≤2 行、封面文字 ≤16 字、标签 4-8 个；标题给出四维评分（搜索匹配/点击吸引/可信度/营销风险）；命中合规禁区词自动标红并阻止进入待发布 |

### 3.2 小红书获客专员（代号：阿盈）

| 项 | 内容 |
|---|---|
| 职责 | 从导入/采集的帖子与评论中识别潜在客户 → 需求分类 → 意向评分 → 写判断理由 → 生成拟回复 → 推送待确认 → 跟踪跟进结果 |
| 输入 | 原始评论数据（第一阶段：CSV/粘贴导入；第二阶段：本地轻量采集）、关键词任务、知识库（服务范围/报价口径/销售话术/合规禁区）、自家笔记的评论区 |
| 输出 | 每日 10-20 条结构化 Lead（含全部 12 个约定字段）、每条高意向线索一份拟回复、每日线索简报（可推送）、跟进结果记录 |
| 状态 | idle → working（判定中）→ waiting_approval（高意向待确认）→ done → blocked |
| 禁止行为 | 不自动发送任何私信/评论；不编造用户信息；评分 0-39 的线索不入库（只计数）；不对"扣1/666"类引导评论直接判高分（需结合帖子上下文）；不生成骚扰式、模板感过重的回复 |
| 质量规则 | 每条线索必须有 post_url 和 comment_text 原文（无来源不入库）；reason 必须引用评论原文中的具体证据；lead_score 分档：80-100 强意向/60-79 中意向/40-59 弱意向仅记录；拟回复 ≤80 字、必须针对评论内容、不含二维码/微信号等违规引流词；同一用户 7 天内去重 |

### 3.3 协作规则（第一个员工闭环）

- 阿禾发布的笔记登记 post_url 后，其评论区数据导入时自动标记 `source=own_post`，阿盈优先处理；
- 阿盈每周向阿禾反馈：哪些 detected_need 出现最多、哪些内容带来的线索评分最高；
- 阿禾的周复盘必须包含"下周继续写 / 停止写"两个清单，写入选题库权重。

## 4. 端到端流程

```
① 公司资料录入（一次性，30 分钟）
   老板填写：公司名/城市/业务范围/报价口径/服务流程/常见问答/禁区
   → 存入知识库五大类。资料完整度 < 60% 时，两位员工处于 blocked 状态并明确提示缺什么。

② 内容生成（每天，自动/手动触发）
   阿禾读取知识库 + 选题库 + 上期复盘 → 生成 3 篇 ContentDraft
   → 进入"待你确认"队列。

③ 人工发布（每天，10 分钟）
   老板在内容详情页：编辑 → 通过/驳回 → 复制发布清单到手机 → 在小红书 App 人工发布
   → 回填 post_url，笔记状态变为"已发布"。

④ 评论/线索发现（每天）
   第一阶段：老板把自家笔记评论区 + 关键词搜索结果（CSV/粘贴）导入
   第二阶段：本地执行器半自动采集（见第 8 节）
   → 阿盈逐条判定：detected_need / lead_score / reason / suggested_reply。

⑤ 拟回复确认（每天，10 分钟）
   评分 ≥80 的线索进入"待你确认"队列 → 老板查看原评论 + 理由 + 拟回复
   → 通过（复制回复文本，人工去小红书发）/ 编辑后通过 / 忽略。

⑥ 人工跟进
   老板在线索详情标记跟进状态：待跟进 → 已回复 → 已私信 → 已转化 / 无效
   每次状态变更写 ActivityLog。

⑦ 结果复盘（每周）
   数据页录入各笔记 views/likes/collects/comments/new_leads
   → 系统生成 WeeklyReview：内容-线索归因表、继续写/停止写清单、下周关键词建议
   → 结论自动喂给阿禾的下一轮选题。
```

关键设计：③⑤两步是全流程仅有的两个人工卡点，每步控制在 10 分钟内，这是"用户愿意每天打开"的前提。

## 5. 数据模型草案

单机 SQLite，全部含 `id`、`created_at`、`updated_at`。核心表 10 张：

```
Agent            数字员工
  code(content_op|lead_gen) name avatar status(idle|working|waiting_approval|done|blocked)
  daily_quota today_done_count persona_note

Task             任务（关键词任务/日常任务）
  agent_id type(content_daily|lead_scan|weekly_review) title
  config_json(keywords,city,industry,daily_count,style)
  status(pending|running|waiting_approval|done|failed) result_summary run_at

KnowledgeItem    知识条目
  category(company|service|sales|platform|compliance)
  title content tags source(user|system_seed) enabled

ReferencePost    参考爆款
  platform keyword post_title post_url author_name like_count collect_count comment_count
  post_type cover_text post_summary content_structure hook_sentence pain_point solution
  call_to_action comment_feedback imitate_level(高|中|低) risk_note

ContentDraft     内容草稿
  task_id topic_type(搜索|避坑|流程|案例|对比) title title_variants_json title_scores_json
  cover_text cover_plan_json body tags_json comment_guide publish_note
  status(draft|waiting_approval|approved|rejected|published) reject_reason
  post_url publish_time stats_json(views,likes,collects,comments,new_leads) ref_post_ids_json

Lead             线索（字段与需求约定完全一致）
  platform keyword post_title post_url comment_user_name profile_url comment_text
  detected_need lead_score reason suggested_reply
  source(own_post|keyword_search|manual) source_content_id(可空，关联自家笔记)
  status(new|waiting_approval|approved|replied|messaged|converted|invalid) follow_note

Approval         审批记录
  target_type(content|lead_reply) target_id action(approve|reject|edit_approve)
  before_json after_json operator note

PushNotification 推送（第一版落库+页面提醒，通道预留）
  channel(inapp|wecom_bot|workbuddy) title body payload_json status(pending|sent|read)

ActivityLog      活动流
  actor(agent_code|user) action target_type target_id detail created_at

WeeklyReview     周复盘
  week_start summary_md best_content_ids_json best_leads_count
  keep_writing_json stop_writing_json next_keywords_json content_lead_map_json
```

约束：
- Lead 的 `post_url` + `comment_text` 建唯一索引（防重复入库）；`comment_user_name` 7 天窗口去重；
- ContentDraft 从 `waiting_approval` 到 `published` 必须存在对应 Approval 记录；
- 一切删除为软删除，ActivityLog 只增不改。

## 6. 知识库结构

五大类，每类给种子模板，用户改而不是从零写（对抗"不愿填资料"）：

| 类别 | 内容 | 种子示例 |
|---|---|---|
| 公司知识 | 公司名、城市、成立年份、团队规模、资质、服务过的行业、差异化一句话 | "XX 财税，东莞，8 年，服务 300+ 电商个体户" |
| 服务知识 | 业务清单及口径：注册公司/代理记账/税务筹划/工商变更/注销；每项的流程、周期、所需材料、价格区间（区间而非承诺价） | "东莞注册公司：3-5 个工作日，需身份证+地址材料，代办费 X-Y 元" |
| 销售知识 | 常见问题标准答复、跟进话术、不同客户类型（电商个体户/初创/自由职业者）的沟通要点、报价原则 | "客户问'最低多少钱'时：先问行业和开票需求，不直接报底价" |
| 平台知识 | 小红书排版规则、发布时间建议、标签策略、标题类型库、爆款结构模板（系统内置，随版本更新） | "封面文字 ≤16 字；晚 8-10 点发布；每篇 4-8 个标签" |
| 合规禁区 | 违禁词表（最便宜/保过/100% 退税/绝对合规…）、不可承诺事项、财税行业红线、小红书社区规范要点 | "不得承诺'包拿执照'；不得出现'加微信'字样" |

机制：
- 公司+服务两类为"必填最小集"（约 10 个字段），完整度进度条展示；不足 60% 员工不开工；
- 平台知识和合规禁区由系统内置、只读+可补充；
- 生成内容/拟回复时，合规禁区作为硬约束注入 prompt，并在产出后二次做违禁词扫描。

## 7. 审批中心设计

不设独立页面，作为办公室首页的"待你确认"队列，按类型分组：

| 需要确认的事项 | 触发条件 | 用户可做的动作 |
|---|---|---|
| 内容草稿 | 每篇生成后必审（100%） | 通过 / 编辑后通过 / 驳回（写原因，理由回流选题库） |
| 高意向线索拟回复 | lead_score ≥ 80 | 通过并复制回复 / 编辑后通过 / 忽略 |
| 中意向线索 | 60-79 分 | 不强制审批，进入线索池"跟进池"标签，随时可看 |
| 合规拦截件 | 内容或回复命中违禁词 | 只能"编辑后通过"或驳回，不能直接通过 |
| 员工 blocked | 知识库不完整 / LLM 配置缺失 | 跳转补齐 |

原则：
- **人工确认是产品特性不是摩擦**：确认页必须并排展示"产出 + 依据"（内容旁边放参考爆款，回复旁边放评论原文和 reason）；
- 一切"发出去"的动作（发布笔记、回复评论、发私信）都发生在小红书 App 内、由人执行，OneDesk 只提供复制好的最终文本；
- 每次审批落 Approval 表，编辑差异保留 before/after，未来用于优化生成质量。

## 8. 本地执行器设计

### 阶段 A（第 1-3 周，本 MVP）：手动导入，零风控风险

- 线索输入三种方式：
  1. **粘贴导入**：把小红书评论区文本整段粘贴，系统解析出 用户名/评论内容 列表；
  2. **CSV 导入**：给出模板（platform, keyword, post_title, post_url, comment_user_name, profile_url, comment_text）；
  3. **单条手录**：详情表单。
- 爆款参考同理：粘贴笔记正文 + 填基本字段，阿禾负责拆解结构。
- 优点：不碰平台风控、当天可用、数据质量由人保底。这是验证"AI 判定 + 拟回复 + 人工确认"这个核心价值的最短路径。

### 阶段 B（第 4 周起，可选交付）：本地轻量采集器

- 形态:独立的本地 Node 脚本（Playwright 驱动用户日常 Chrome 的持久化 profile），不是云端爬虫;
- 行为约束：
  - 只做用户手工也会做的事：搜索关键词 → 打开帖子 → 读取公开评论；
  - 限速：每关键词每天 ≤30 帖，随机间隔 8-20 秒，只在用户设定的时间窗运行；
  - 只读不写：永不自动点赞/评论/关注/私信。
- **登录与验证码**：一律人工。执行器检测到登录态失效/出现验证码时立即暂停，弹系统通知（第一版本地通知，后续接微信推送），用户在同一浏览器窗口人工完成后点"继续"；绝不做打码、绝不自动重试登录。
- **风控与失败处理**：
  - 检测到滑块/异常页/内容为空 → 立即停止当天任务，标记 Task=failed，记录截图与原因；
  - 连续 2 天失败 → 自动降级回阶段 A 手动导入，并在办公室首页提示；
  - 所有采集原始数据落本地工作区目录（`~/OneDesk/workspace/`），CSV 永远可导出，用户数据不出本机。

## 9. UI 草图（文字描述）

整体气质：**一张桌子**。米白纸感底色 + 墨色文字 + 少量朱砂红点缀；标题用衬线中文字体（思源宋体），正文无衬线；两位员工用简洁插画头像而非像素游戏风；无金币、无经验值、无抽卡。参考 claw-empire 的"员工状态 + 任务流 + 活动流"骨架，但视觉走安静的文房质感，不走游戏化。

1. **办公室（首页）**
   - 顶部：公司名 + 日期 + 一句今日摘要（"阿禾备好了 3 篇笔记，阿盈找到 14 条线索，其中 5 条强意向"）；
   - 中部左：两张"工位卡"——员工头像、状态、今日进度（3/3 篇、14/20 条）、一个主按钮（开工/查看产出）；
   - 中部右：**待你确认**队列（内容 3、拟回复 5、拦截 1），点击进入审批视图；
   - 底部：活动流时间线（"09:12 阿盈完成 20 条评论判定，入库 14 条"）。
   - 主要操作：触发生成、进入审批、查看产出。

2. **员工页**：每位员工一张档案卡（职责/禁止行为/质量规则，即"岗位说明书"，可读不可改）+ 今日任务 + 历史产出列表 + 近 7 天产出统计。主要操作：调整每日配额、暂停/恢复。

3. **任务页**：任务列表（类型/关键词/状态/结果摘要），新建关键词任务表单（行业、城市、业务、目标客户、内容风格、每日数量）。主要操作：新建、手动运行、查看失败原因。

4. **线索页**：
   - 列表：评分徽章（红 80+/橙 60-79/灰 40-59）、需求分类、评论摘要、来源标签（own_post/搜索）、状态；筛选器：评分段/需求/状态/关键词/日期；
   - 详情抽屉：左侧评论原文 + 帖子链接 + 用户主页链接，右侧 detected_need、lead_score、reason（引用原文证据）、suggested_reply（可编辑、一键复制）、跟进状态按钮、跟进备注；
   - 顶部按钮：导入线索（粘贴/CSV/手录）、导出 CSV。

5. **内容页**：四个标签（选题库/草稿/待发布/已发布）。
   - 草稿详情：手机预览样式（封面文字卡片 + 正文排版），右侧标题备选及四维评分、标签、评论引导、发布注意事项、参考爆款折叠区；
   - 已发布：回填 post_url 与数据入口；
   - 主要操作：编辑、通过、驳回、复制发布清单、导出。

6. **数据页**：本周发布记录表（可编辑 views/likes/collects/comments/new_leads）、内容-线索归因表（哪篇笔记带来几条线索、平均分）、周复盘卡片（继续写/停止写/下周关键词）。主要操作：录入数据、生成周复盘。

7. **知识库页**：五类左侧目录，右侧条目卡片编辑；顶部完整度进度条 + "还差什么"提示。主要操作：编辑条目、启用/停用。

8. **设置页**：公司基础信息、LLM 配置（服务商/BaseURL/Key/模型，支持 OpenAI 兼容接口）、工作区目录、推送配置（第一版展示"即将支持"）。

## 10. 验收标准

**内容质量**
- 每天生成 3 篇完整笔记（标题+封面文字+正文+标签+评论引导+发布注意事项），缺一项算不合格；
- 违禁词拦截率 100%（内置词表范围内）；
- 老板"不改或小改（<30% 文字改动）即通过"的比例 ≥ 60%（用 Approval 的 before/after 计算）；
- 连续 7 天稳定产出，无空窗日。

**线索质量**
- 每天入库 10-20 条线索，100% 带 post_url、comment_text、reason；
- 80+ 分线索中，老板确认"值得跟进"的比例 ≥ 70%（approve+edit / 全部高分件）；
- "扣1/666"类引导评论误判为高分的比例 ≤ 10%；
- 同用户重复入库率 ≤ 5%。

**使用频率**
- 老板连续 7 天每天打开办公室页并完成两个审批队列；
- 每日人工操作总时长 ≤ 25 分钟（审批 20 + 导入 5）；
- 7 天内至少发布 15 篇笔记、跟进 20 条线索。

**复盘有效性**
- 每周生成 1 份 WeeklyReview，含内容-线索归因；
- 至少 1 条"停止写"结论被下一周选题实际执行；
- 第 2-4 周出现 ≥1 条 status=converted 的线索（终极信号）。

**MVP 成立的一句话判据**：用户不再问"今天发什么、去哪找客户"，而是每天打开 OneDesk 花 20 分钟做确认。

## 11. 4 周开发计划

技术栈：Node.js 22 + Express + better-sqlite3（本地单文件库）+ Vite + React + TypeScript + Tailwind；LLM 走 OpenAI 兼容接口（DeepSeek/通义/Kimi 均可配），无 Key 时进入内置模板的演示模式。单仓库、本地运行、零云依赖。

| 周 | 目标 | 交付物 | 测试方法 | 明确不做 |
|---|---|---|---|---|
| W1 | 骨架 + 知识库 + 数据模型 | 可运行的 app：8 页导航、SQLite 全表、知识库五类 CRUD + 种子模板 + 完整度、设置页 LLM 配置 | 手工走查：填完知识库 → 完整度到 100%；重启数据不丢 | 不做采集、不接真实 LLM 生成 |
| W2 | 内容闭环 | 阿禾可用：关键词任务 → 生成 3 篇草稿（LLM+演示模式双通道）→ 审批 → 发布清单复制/导出 → 回填 post_url；违禁词扫描 | 用真实财税知识库跑 3 天，每天人工评审 3 篇是否"可直接发" | 不做封面图生成、不做爆款自动采集 |
| W3 | 线索闭环 | 阿盈可用：粘贴/CSV 导入 → AI 判定评分 + reason + 拟回复 → 高分进审批 → 跟进状态流转 → CSV 导出；own_post 关联；办公室首页 + 活动流完整 | 准备 50 条真实小红书评论样本（含 10 条"扣1"干扰项），核对评分分布与误判率 | 不做手机推送、不做自动采集 |
| W4 | 复盘闭环 + 打磨 | 数据录入 + 内容-线索归因 + 周复盘生成并回流选题；视觉打磨（纸感主题、手机预览）；（余力）阶段 B 采集脚本原型 | 连续 5 天真实使用一遍完整闭环（②→⑦），记录每日操作时长是否 ≤25 分钟 | 不做多账号、不做企业微信推送正式版 |

每周五输出一页复盘：完成/未完成/风险，给何老师演示一次可点的版本。

## 12. 最大风险与规避

| 风险 | 表现 | 规避策略 |
|---|---|---|
| 平台规则 | 采集触发风控、账号被限流封禁 | MVP 阶段只做手动导入（阶段 A）；阶段 B 只读、限速、人工登录、失败即停；产品内永不出现自动私信/评论；文档和 UI 中明确合规边界 |
| 数据质量 | 导入评论格式乱、字段缺失导致判定垃圾进垃圾出 | 无 post_url/comment_text 不入库；导入时即时预览解析结果让用户纠错；每条线索保留原文，判定错了可人工改分并记录 |
| 模型幻觉 | 编造价格、承诺结果、虚构案例、reason 与评论无关 | 知识库口径作为唯一事实来源注入 prompt；产出后跑违禁词+承诺句式扫描；reason 必须含评论原文引用（校验子串命中）；价格只允许输出知识库中的区间 |
| 过度自动化 | 用户期待"全自动获客"，越过人工卡点 | 产品叙事定为"数字员工做 80%，老板做 20% 的确认"；发布/回复动作根本不提供自动通道（不是开关，是没有）；审批页展示依据以建立"确认是有价值的"认知 |
| 过度游戏化 | 装修办公室/养成玩法冲淡工具价值 | MVP 视觉走安静的文房质感；无金币/等级/抽卡；游戏化元素冻结到愿景阶段再评估 |
| 用户不愿填公司资料 | 知识库空转，生成内容空洞 | 五类知识全部提供财税行业种子模板，用户"改"而非"写"；必填最小集控制在 10 个字段/30 分钟；完整度不足时员工 blocked 并具体说缺哪条，把填写压力转化为明确的下一步 |
| （补充）LLM 成本与稳定 | 国内网络/Key 失效导致每天开工失败 | OpenAI 兼容接口可切换服务商；失败自动降级演示模板并明确提示；生成任务有重试与结果缓存 |

## 附：与原设想的三处调整（及理由）

1. **审批中心并入办公室首页**：老板每天只想打开一个页面。独立审批页会把 20 分钟的事变成多页跳转。
2. **每日线索目标定为"10-20 条入库，5-8 条高分"而非固定 20 条**：手动导入阶段数据量受输入限制，宁可少而准，防止为凑数放低评分线（数据质量风险第一）。
3. **采集器移到第 4 周且为可选交付**：核心假设是"AI 判定 + 拟回复 + 人工确认"有价值，这个假设用手动导入即可验证；采集只是降本手段，不值得为它承担第 1 周就碰风控的风险。
